# 可信模块插件架构

Smart Design 的插件机制定位为可信模块化产品平台，不是第三方插件市场。插件用于按客户、交付包或部署 Profile 灵活组合能力；权限只负责模块启用后的访问控制。

## 职责边界

- 主平台内核：用户、权限、项目、标准、图纸、全局布局、插件运行时、插件状态管理。
- 插件运行时：校验 manifest、同步权限、执行 migration、挂载前后端入口、记录生命周期状态。
- 插件实现：插件自己的页面、API、数据库 schema、migration、测试和业务文档。
- 客户配置：决定启用哪些插件、插件设置值、菜单显隐、字段和流程差异。

主平台不应该写死某个插件的业务页面或 API。主平台只提供扩展点，例如页面路由、导航、项目详情动作、设置区块、报表定义、导入模板、后端 API mount 和数据库 migration。

## 插件不是权限

- 插件回答：系统有没有这个能力。
- 配置回答：这个能力在当前客户环境下怎么表现。
- 权限回答：谁可以使用这个能力。

例如一个客户专属分析模块：

- 客户不需要该分析能力：不启用对应插件。
- 客户需要该分析能力但只允许项目经理使用：启用插件后用项目权限控制。
- 客户 A/B 的字段、状态、报表不同：优先用插件配置或客户扩展插件，不要在主平台写客户分支。

## 路由约定

前端插件页面默认挂在插件命名空间下：

```text
/plugins/{pluginId}/*
```

后端插件 API 默认挂在：

```text
/api/plugins/{pluginId}/{apiPrefix}/*
```

这样主平台只知道“插件可以贡献页面和 API”，不需要预先知道每个插件的具体业务路由，也能避免客户模块之间抢占全局路径。

## Manifest 是平台契约

`plugin.json` 是平台和插件之间的唯一声明入口，至少应覆盖：

- `id/version/min_host_version`
- `module`
- `dependencies`
- `frontend.entry/routes/navigation/slots`
- `backend.entry/api_prefix`
- `permissions`
- `default_role_grants`
- `database.schemas`
- `migrations`

权限定义和默认角色授权由主平台 runtime 根据 manifest 同步。插件 migration 只负责插件自己的数据库对象，不应承担平台权限目录维护。

## 生命周期

当前生命周期目标：

- `upload`：校验签名和 manifest，存储包内容。
- `install`：同步权限、执行 migration、登记能力、写入安装状态。
- `enable`：先加载插件路由，成功后再标记为 enabled。
- `disable`：卸载内存路由，再标记为 disabled。
- `uninstall`：卸载内存路由、隐藏包能力、保留数据。
- `purge`：按 manifest 声明的 schema 清除插件数据。

后续可继续补 `healthcheck`、异步安装任务、升级计划、回滚策略和插件配置页面。
