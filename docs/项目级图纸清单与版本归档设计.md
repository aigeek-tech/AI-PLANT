# 项目级图纸清单与版本归档设计

## 摘要
- 新增项目级 `document` 子域，作为项目下与 `tag` 并列的一级对象；清单按项目管理，不挂靠 PBS 或 Tag，但可选关联多个 PBS 节点和多个 Tag。
- 采用 `document -> revision -> file` 三层模型：`document_no` 作为逻辑图纸编号，`revision_no` 管修订历史，单个修订下允许挂多个文件。
- 第一版文件处理范围固定为“归档与预览”：使用 S3/MinIO 兼容对象存储、短期签名上传/下载/预览、保留上传目录相对路径；CAD/Office 等常规文件预览接入 kkFileView，不做内容解析。

## 领域与接口
- `document_type`：全局可配置字典，字段固定为 `code/name/description/status/allowed_extensions`；由标准模块新增“文档类型”配置页维护。
- `document_type_attribute`：按类型定义属性模板，字段沿用现有属性体系风格 `code/name/value_type/is_required/enum_options/sort_order/description`。
- `document`：字段固定为 `project_id/document_no/title/document_type_id/discipline/attributes/current_revision_id/status/metadata`，唯一约束 `(project_id, document_no)`，`status` 固定为 `active|archived`。
- `document_revision`：字段固定为 `document_id/revision_no/state/is_current/issued_at/change_summary`，唯一约束 `(document_id, revision_no)`，`state` 固定为 `draft|issued|void`，同一图纸只允许一个 `is_current=true`。
- `document_file`：字段固定为 `revision_id/file_role/original_filename/relative_path/storage_provider/bucket/object_key/mime_type/size_bytes/checksum_sha256/etag/preview_mode/status`；`file_role` 固定为 `primary|source|attachment|reference`，同一修订仅允许一个 `primary` 就绪文件；`status` 固定为 `pending_upload|ready|upload_failed|deleted`。
- `document_pbs_link` 与 `document_tag_link`：使用两张关联表表达图纸与 PBS/Tag 的可选多对多关系，不使用多态外键。
- 文件目录不单独建目录树实体；用 `relative_path` 保存上传时的相对目录，用 `object_key` 保存对象存储真实路径。对象键统一为 `projects/{projectId}/documents/{documentId}/revisions/{revisionId}/{fileId}-{sanitizedFilename}`。
- 项目 API 新增：
  - `GET/POST /api/projects/{projectId}/documents`
  - `GET/PATCH /api/projects/{projectId}/documents/{documentId}`
  - `GET/POST /api/projects/{projectId}/documents/{documentId}/revisions`
  - `PATCH /api/projects/{projectId}/documents/{documentId}/revisions/{revisionId}`
  - `POST /api/projects/{projectId}/documents/{documentId}/revisions/{revisionId}/files/initiate`
  - `POST /api/projects/{projectId}/documents/{documentId}/revisions/{revisionId}/files/{fileId}/complete`
  - `GET /api/projects/{projectId}/documents/{documentId}/revisions/{revisionId}/files/{fileId}/access-url`
- `POST /documents` 请求体固定包含 `document_no/title/document_type_id/discipline/attributes/pbs_node_ids/tag_ids/status`；`POST /revisions` 固定包含 `revision_no/state/issued_at/change_summary/set_as_current`；`files/initiate` 固定包含 `filename/file_role/relative_path/content_type/size_bytes/checksum_sha256`，返回 `file_id/upload_url/upload_headers/expires_at/object_key`。
- 列表接口默认返回当前修订扁平摘要：`current_revision_no/current_revision_state/file_count/primary_file_name/linked_pbs_count/linked_tag_count`，并支持 `keyword/document_type_id/discipline/pbs_node_id/tag_id/status` 过滤与分页。

## 实现变更
- 后端继续沿用 FastAPI + psycopg 原生 SQL 结构，新增独立 repository/service 模块处理文档域，不复用 `tag_import` 草稿表。
- 新增对象存储适配层，第一版只实现 S3/MinIO 兼容适配器；配置项固定为 `S3_ENDPOINT/S3_REGION/S3_BUCKET/S3_ACCESS_KEY/S3_SECRET_KEY/S3_PRESIGN_TTL_SECONDS/S3_KEY_PREFIX`，凭证只保留在服务端。
- 前端在项目详情页新增“图纸清单”视图，与现有 PBS/位号并列；页面包含清单表格、图纸详情抽屉、修订时间线、多文件上传对话框、文件预览/下载入口。
- 前端增加文档类型配置页到标准模块；项目页只消费类型配置，不允许项目内临时创建新类型。
- 预览策略固定为：前端统一打开系统内预览页；启用 kkFileView 时由 kkFileView 处理 CAD、Office、PDF、图片、文本等常规格式，未启用时 PDF、图片、文本回退到浏览器 `inline` 预览，其余格式只提供下载；内容解析、图签抽取、元数据抽取全部留作后续异步扩展。
- 本地开发环境在 `docker-compose` 增加 MinIO 与 bucket 初始化；生产环境继续通过 S3 兼容参数切换，不改业务代码。

## 测试计划
- 数据库迁移测试覆盖唯一约束、当前修订唯一约束、主文件唯一约束、PBS/Tag 级联删除行为。
- API 测试覆盖图纸创建、重复编号拦截、修订切换、修订历史保留、多文件上传完成、签名 URL 失效或对象不存在、文件软删除、列表过滤分页。
- 前端测试覆盖项目详情页清单展示、按类型和关键字过滤、创建图纸、为同一图纸新增修订、一次修订上传多文件并保留相对目录、PDF inline 预览、非预览文件下载。
- 回归测试覆盖现有项目管理、PBS 节点、Tag 创建与 Tag 导入流程，确保新增文档子域不破坏现有工程页加载和标准页行为。

## 假设与默认
- 图纸清单是项目级业务对象，与位号并列，不强制挂在 PBS 或 Tag 下。
- `document_no` 在同一项目内唯一；一个图纸始终保留完整修订历史，且始终最多一个当前修订。
- 每个修订允许多个文件，至少支持一个 `primary` 发放件；源文件如 DWG、Excel 归类为 `source`。
- 第一版不做 CAD/PDF 内容解析、OCR、图签抽取或自动编号，只做元数据管理、版本管理、对象存储归档和基于 kkFileView/浏览器的在线预览。
- 目录需求通过 `relative_path` 满足，不在 v1 引入独立“目录树/文件夹”实体。
